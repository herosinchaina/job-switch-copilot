## 一、定位与数据范围

**差异化价值(相对模块三知识库)**:知识库问"记住没"(自评翻卡),错题本问"现在能不能答对"(AI 重做评分),并提供薄弱分布统计。方案 **A+B**:攻克闭环 + 错因洞察。

- **数据范围**:仅 `source ∈ {interview, deepdive}` 的条目算错题。手动条目(`source=manual`)是主动积累,不算错题,留在知识库。
- **状态方案(C)**:新增 `knowledge_attempts` 重做尝试表(存每次重做的答案/分数/反馈,支持重做历史)+ `knowledge_items` 加一列 `conquered_at`。
- **待攻克** = `source∈{interview,deepdive} 且 conquered_at IS NULL`。
- **已攻克** = `conquered_at IS NOT NULL`。
- **攻克后不自动打回**:`conquered_at` 一旦置位就保留(代表"曾经攻克"),避免反复横跳;攻克后仍可再重做,记录照样入 attempts 表。

**硬约束沿用**:流畅性、稳定性、安全性优先。AI 输出 zod 校验、非法 JSON 重试→降级、SQL 参数化、限并发队列(≤2)、CLI 超时 kill、写用事务。

## 三、AI 重做评分(简化独立评分)

- 走现有 `AiProvider` 适配层,**一次无状态调用**(不开 session,不还原原始 JD/项目上下文)。
- **Prompt 只喂三样**:题目 `question` + 参考答案 `reference`(可能为 null,提示 AI 自行判断)+ 用户新答案。要求返回 `{score, comment, gaps}` JSON。
- **AI 输出 zod 校验**(`AttemptGradeRawSchema`),非法 JSON → 重试 → 降级报错,不静默崩。
- **verdict 由后端算**,不信任 AI:`verdict = score >= CONQUER_THRESHOLD ? 'pass' : 'fail'`(统一 60 及格线)。
- **攻克置位**:`verdict==='pass'` 且当前 `conquered_at IS NULL` 时,事务内一并写 `conquered_at = datetime('now')`。已攻克的再重做只追加 attempt,不改 `conquered_at`。
- 受限并发队列(≤2)、CLI 超时 kill 沿用现有基础设施。
- **模型**:用 CLI 默认模型(评分类,非 LeetCode 引导,不强制 sonnet)。

## 四、后端 API

静态段先于 `/:id` 注册(沿用现有约定)。建议并入现有 `knowledge` 路由文件或新建 `routes/knowledge-book.ts` 挂同一 `/api` 前缀。

```
GET  /api/knowledge-book            待攻克/已攻克列表
                                    ?status=pending|conquered(默认 pending)
                                    &source=interview|deepdive & tag=xxx
GET  /api/knowledge-book/stats      错因洞察:总数/待攻克/已攻克、按来源计数、
                                    按标签薄弱计数、近 7 天攻克数
GET  /api/knowledge/:id/attempts    某条的重做历史(倒序)
POST /api/knowledge/:id/attempt     body: { answer } → AI 评分 → 返回 { feedback, conquered, attempt }
```

### repo 新增

- `listBookItems(db, { status, source, tag })` — 按 `source∈{interview,deepdive}` + `conquered_at` 是否为空 + 可选来源/标签过滤,附带每条 `attemptCount`。
- `bookStats(db)` — 聚合:`{ total, pending, conquered, bySource:[{source,count}], byTag:[{tag,count}](薄弱=未攻克), conqueredLast7Days }`。
- `listAttempts(db, itemId)` — 倒序返回该条所有重做记录。
- `createAttempt(db, { itemId, answer, score, feedback, source })` — **事务**:插 attempt 行;若 `verdict==='pass'` 且 `conquered_at IS NULL` 则置 `conquered_at`。返回新 attempt + 是否本次攻克。

> AI 调用在 route 层组织(取 item → 组 prompt → 调 AiProvider → zod 校验 → 算 verdict → 调 `createAttempt`)。repo 只做纯 DB,不含 AI。

### 入参健壮性(沿用知识库经验)

- `status` 非 `pending|conquered` → 视为 `pending`(或 400,取一致策略:400 更严格,采用 400)。
- `:id` 非正整数 / 条目不存在 → 404。
- `answer` 空串 → 400(zod)。
- 非法来源条目(source=manual)调 attempt → 400「该条目不是错题」。

## 五、前端

### 导航(`apps/web/src/App.tsx`)

新增 `view: 'errorbook'` + 导航项「错题本」,图标 `Target`(lucide)。无前置门槛直接渲染(数据为空时给空态引导)。

### 页面(`apps/web/src/pages/ErrorBook.tsx`),三个 Tab

1. **待攻克**:差题卡片(题目 / 来源徽章 / 标签 / 重做次数)。点开 →
   - 展示原题 + 参考答案(Markdown)。
   - 文本框写新答案 → 「让 AI 评分」(loading 态)。
   - 返回后展示 `score`、`verdict`(通过/未通过徽章)、`comment`、`gaps` 列表。
   - `pass` → 卡片标「已攻克 🎉」,可留在原地或刷新移出待攻克列表。
2. **已攻克**:攻克归档列表,可展开看重做历史(`listAttempts`)。
3. **洞察**:统计面板
   - 顶部指标卡:总错题 / 待攻克 / 已攻克 / 近 7 天攻克。
   - 按来源分布(面试 vs 深挖)。
   - 按标签的薄弱分布(未攻克计数,条形/排行),帮用户看清反复栽跟头的知识点。

复用 `Markdown`、`Card/Button/Badge`、`Mastery`(如需)等现有组件。
