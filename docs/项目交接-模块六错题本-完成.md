## 一、项目定位与架构(已定型,勿改)

**产品**:AI 求职操作系统(个人自用真实产品,方案 A)。需求文档:`data/AI_Interview_OS_Detailed_Product_Prompt.docx`(已 gitignore)。

**架构**:
```
React 前端 (Vite+TS+Tailwind)  ←HTTP+SSE→  本地 Node 后端 (Express)  ←→  claude CLI 子进程
                                              SQLite (node:sqlite, --experimental-sqlite)
```
- **monorepo**:`apps/web` + `apps/server` + `packages/shared`(共享 TS 类型/zod 模型)
- **AI 引擎**:全部走 `AiProvider` 适配层 → `ClaudeCliProvider`(`spawn('claude', [...])`,`shell:false`)。复用本机 Claude CLI,不填 API Key。
- **多轮会话**:`ClaudeCliProvider` 支持 `startSession/continueSession`;DB 存权威 Q&A;`--resume` 失败自动降级"拼历史"无状态模式。
- **AI 老师(仅 LeetCode 引导)固定 `--model sonnet`**,其余用 CLI 默认模型。

**三大硬约束**:流畅性、稳定性、安全性优先。
- 安全:`shell:false` 数组传参、仅绑 `127.0.0.1`、上传限制、AI 输出 zod 校验、SQL 参数化、限并发队列(≤2)、CLI 调用超时 kill、CLI 启动自检横幅。
- 稳定:AI 非法 JSON → 重试 → 降级报错,不静默崩;SQLite 写用事务。

**关键文件**:DB schema `apps/server/src/db/connection.ts`;repo `apps/server/src/db/repo.ts`。差题打标字段 `is_weak`(interview_turns 和 project_deepdive_turns 都有)。

## 四、模块六 错题本 —— 已完成实现细节

**定位**:知识库中来自模拟面试/项目深挖的差题的**攻克 + 洞察中心**。知识库问"记住没"(自评翻卡),错题本问"现在能不能答对"(AI 重做评分),并提供薄弱分布统计。方案 **A+B**:攻克闭环 + 错因洞察。

**数据范围**:仅 `source ∈ {interview, deepdive}` 的条目算错题。手动条目留在知识库,不进错题本。

**状态方案(C)**:
- `knowledge_items` 新增 `conquered_at TEXT` 列(幂等 ALTER)。
- 新增 `knowledge_attempts` 表(重做历史):`id / item_id / answer / score / feedback_json / created_at` + 索引 `ix_ka_item`。
- 待攻克 = `source∈{interview,deepdive} 且 conquered_at IS NULL`;已攻克 = `conquered_at IS NOT NULL`。**攻克后不打回**。

**AI 重做评分(简化独立评分)**:
- 走 `AiProvider` 一次无状态调用,prompt 只喂三样:题目 + 参考答案(可 null)+ 新答案。
- AI 只输出 `{score, comment, gaps}`(`AttemptGradeRawSchema` zod 校验,`completeJson` 2 次重试→抛错,不静默假通过)。
- **verdict 后端算,不信任 AI**:`score >= CONQUER_THRESHOLD(60) ? 'pass' : 'fail'`(统一 60 及格,不再分来源阈值)。
- pass 且 `conquered_at IS NULL` → 事务内置 `conquered_at`;已攻克再重做只追加 attempt。

**已落地文件**:
- `packages/shared/src/knowledge.ts` — `CONQUER_THRESHOLD=60`、`AttemptGradeRawSchema`、`KnowledgeAttemptFeedbackSchema`(含 verdict)、`KnowledgeAttemptSchema`;`KnowledgeItemSchema` 加 `conqueredAt`
- `apps/server/src/db/connection.ts` — 加列 + 建表 + 索引
- `apps/server/src/db/repo.ts` — `createAttempt`(事务+攻克置位)/`listAttempts`(倒序)/`listBookItems`(排除 manual)/`bookStats`(byTag 仅未攻克);`rowToKnowledgeItem` 补 `conqueredAt`;`exportAll` 加 `knowledgeAttempts`
- `apps/server/src/services/errorbook.ts` + `apps/server/src/prompts/errorbook-grade.txt` — `gradeAttempt`
- `apps/server/src/routes/errorbook.ts` — 4 端点(见下),`index.ts` 挂载
- `apps/web/src/api.ts` — `listErrorBook`/`errorBookStats`/`listAttempts`/`submitAttempt`
- `apps/web/src/pages/ErrorBook.tsx` — 三 Tab(待攻克/已攻克/洞察);`App.tsx` 加「错题本」导航(`Target` 图标)

**后端 API**(均挂 `/api`,静态段先于含参段):
```
GET  /error-book?status=pending|conquered&source=&tag=   列表(status 非法 400)
GET  /error-book/stats                                    洞察统计
GET  /error-book/items/:id/attempts                       重做历史(条目不存在 404)
POST /error-book/items/:id/attempt  {answer}              AI 重做评分(manual 400 / answer 空 400 / 不存在 404)
                                                          → {feedback, conquered, attempt}
```

## 五、已知非阻塞项(审查记录,后续可优化)

1. **byTag 洞察依赖手动打标**:`importWeakItem`(模块三)导入差题时 `tags` 硬编码为 `[]`,故"薄弱知识点"面板对真机导入数据默认为空,除非用户手动编辑条目打标。空态已优雅处理。若要让洞察更有价值,可在导入时自动打标(未来优化)。
2. **catch 静默吞错**:`ErrorBook.tsx` 的 `load().catch(()=>[])`/`Insight` 的 `.catch(()=>{})` 会把接口错误吞成空态,掩盖真实失败。可后续加错误提示。
3. **时区细节**:`conquered_at` 用 `datetime('now')`(UTC),`conqueredLast7Days` 用 `date('now','localtime')` 比较,跨午夜/时区可能 ±1 天。影响极小。
4. **schema 冗余**:`knowledge_attempts.score` 列与 `feedback_json.score` 同源冗余,无害。
